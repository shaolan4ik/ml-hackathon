# CHECKLIST перед коммитом решения

## Как использовать

Перед каждым коммитом/обновлением MR пройдитесь по пунктам ниже.  
Если хотя бы один критичный пункт не выполнен, коммит лучше не отправлять.

## 1) Контракты и совместимость

- [ ] RPC-контракты не сломаны: `user`, `event`, `shift`, `prepare`, `ready`, `predict`, `health`, `metrics`.
- [ ] Формат train/validation CSV не изменен (см. `DATA.md`).
- [ ] Изменения совместимы с текущим `REGLAMENT.md`.

## 2) Локальная проверка кода

- [ ] Установлены зависимости: `make install`.
- [ ] Тесты проходят: `make test`.
- [ ] Pre-commit проверки проходят: `make precommit`.
- [ ] Coverage gate соблюден (>= 80% по критичным модулям).

## 3) Готовность к MR

- [ ] В описании MR указано, что именно изменено и зачем.
- [ ] Все зависимости зафиксированы в `pyproject.toml`, а версии зафиксированы в `poetry.lock`
- [ ] В составе решения приложено хотя бы одно из двух:
  - [ ] обученный артефакт модели, совместимый с сервисом;
  - [ ] скрипт/пайплайн обучения, который воспроизводимо готовит модель для сервиса.
- [ ] Если передается скрипт обучения, в MR и документации есть команда запуска и путь к ожидаемым артефактам.
- [ ] Указаны риски/ограничения и способ проверки.
- [ ] Все временные/отладочные фрагменты удалены.
- [ ] Нет лишних файлов в коммите (временные артефакты, локальные дампы и т.д.).

## Полезные команды

```bash
make test
make precommit
make load-test
```
