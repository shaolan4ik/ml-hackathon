# Регламент валидации

## Контракты данных для валидации

### Обучающий датасет (начальная загрузка)

Используются три CSV-файла, которые выдадут вам вместе с заданием:

Эти файлы нужно положить по следующим путям:

- `data/train/user.csv`
- `data/train/shift.csv`
- `data/train/event.csv`

В этих файлах вы найдете следующие таблицы:

- `user.csv`: `location_id`, `is_strict_location`, `id`, `has_mk`
- `shift.csv`: `id`, `start_at`, `location_id`, `task_type`, `employer_id`, `workplace_id`, `need_mk`, `id_differential`, `hours`, `reward`, `capacity`
- `event.csv`: `id`, `shift_id`, `user_id`, `interaction`, `ts`

### Валидационный датасет (дневной цикл оценки)

Для валидации, мы отдельно создадим

- `data/validation/apply.csv` с колонками: `user_id`, `shift_id`, `date`
- `data/validation/event.csv` со схемой событий
- `data/validation/shift.csv` со схемой смен

`apply.csv` используется как целевой файл для расчета качества по дням.

## Как запустить валидацию

Полный запуск eval выполняется командой:

```bash
poetry run python -m hackaton.eval.cli run \
  --host 127.0.0.1 \
  --port 8000 \
  --user-path data/train/user.csv \
  --shift-path data/train/shift.csv \
  --event-path data/train/event.csv \
  --val-apply-path data/validation/apply.csv \
  --val-shift-path data/validation/shift.csv \
  --val-event-path data/validation/event.csv \
  --output-dir artifacts/eval_run \
  --predict-max-concurrency 4 \
  --predict-max-rpm 200
```

Результат валидации: `artifacts/eval_run/eval_report.md`.

Смежные документы:

- `README.md` — общий процесс и быстрый старт.
- `HOW-TO.md` — практический маршрут участника.
- `DATA.md` — форматы train/validation таблиц и примеры.

## Правила расчета метрики

Для каждого дня валидации:

0. Загружаем данные из Train и по API вгружаем в систему по API
1. Перед запуском оценки определенного дня, мы проверяем, что все данные вгружены в систему для оценки. Вызываем `prepare` для первичного расчета/материализации (до 20 минут для первичной оценки по TRAIN данным и до 5-ти минут для обновления по данным предыдущего дня). Если время превышает выделенное, то тест останавливается. Для подготовки мы используем те же ресурсы, что и на Inference. Данные загружаются из специальных файлов с форматом по аналогии с `shift.csv` и `event.csv`.
2. Смены внутри дня мы группируем по емкости (`capacity`), чтобы задать корректное ограничения FPR
3. Обстреливаем вероятности модели (`predict`) для кандидатов. На
4. Отбираем TOP-10 кандидатов на каждую смену по predict-ам вашего решения.
5. Для расчета метрики фиксируем размер пула кандидатов: `pool_size = 10` (константа для всех смен).
6. Применяем ограничение FPR в зависимости от емкости смены (внутри групп по емкости) по формуле:
   - `max_fpr = min(1.0, capacity / 10)`
7. Сравниваем предсказания с фактическими метками из `apply.csv` по `user_id`, записанный на `shift_id` в указанную дату. Если пользователь предсказан корректно - это 1, если нет - 0.
8. Агрегировать метрику дня по группам и считаем среднее значение метрики за день.
9. Собираем данные за последующие дни по шагам 1-8, пока не дойдем до последнего.

Итоговая метрика считается как среднее по всем дням валидации.

По итогам рассчета метрик мы формируем отчет, который содержит:

- Статистика данных на момент замера
- Метрики качества по каждой группе в определенный день
- Метрика качества за день
- Оценка производительности и временные затраты на `prepare` с точностью до 10-ти секунд
- Причины остановки теста, если таковы имеются
- Общая оценка качества
